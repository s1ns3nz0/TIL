# 1. 주요 질문
## 1) 사용자가 URL을 입력했을 때 첫 번째로 만나는 컴포넌트는?
- AWS의 Route53(DNS 서비스)
    + 브라우저에서 입력받은 영어 웹주소를 IP로 변경하여야 서버로 요청 전송 가능
    + 해당 웹주소의 IP로 전환하기 위해서는 DNS 서버에 조회가 필요함
    + 재귀적 DNS 질의 과정 중에 요청한 웹 사이트의 권한있는 네임서버(Route 53)에 조회를 하게 된다.
## 2) HTTPS는 어디에서 종료되는가?
- 주로 웹 서버 앞단에 있는 로드밸런서에서 종료된다.
- 로드밸런서와 서버 간의 통신은 신뢰된 구간으로 간주하며, 로드밸런서와 웹 서버 사이의 구간도 암호화를 수행할 수 있으나 서버에 부담을 줄 수 있다.
## 3) 정적 리소스와 API 요청은 어디서 분기되는가?
- CDN 사용 시, 정적 리소스들은 S3 버킷에서 가져오고 /api 경로를 통해 들어오는 요청은 ALB로 전송
- ALB에서는 URL 경로 기반 라우팅을 기반으로 하여 특정 경로는 정적 파일을 호스팅하는 서버로, 나머지는 API 서버로 전송한다.
## 4) 서버는 어떤 기준으로 확장(AutoScaling)되는가?
- CLoudWatch를 통해 주요 메트릭을 모니터링한다.
    + CPU 점유율: 가장 일반적이며, 특정 수준 이상으로 사용률이 올라가는 경우 확장
    + 요청 수: 특정 시간당 들어오는 요청수가 급증할 때 확장
    + 메모리 사용량: 메모리 특성에 따라 서버의 메모리가 부하가 클 때 확장
## 5) 장애가 발생했을 때 어느 지점에서 감지되는가?
- ELB Health Check: 로드 밸런서가 주기적으로 서버에 헬스체크 신호를 보내는데, 응답이 없는 경우 서버를 트래픽 대상에서 제외한다.
- CloudWatch Alarms: 서버의 리소스 부족, 에러 발생 등 모니터링 중인 메트릭을 통해 트리거되도록 설정한 알람들을 통해 확인 가능하다.
- Route 53 Health Check: 특정 IP 주소, 도메인에 주기적으로 요청을 보내서 확인
    + HTTP, HTTPS: 특정 URL로 접속해서 응답을 기다린다. 
    + TCP: EC2 인스턴스의 특정 포트가 열려 있는 지 확인
